STEP 1
有 preview_html？最高优先
成品积木直接拼上
🧱 成品块 — 设计师已经拼好的完整积木，直接放到底板上，不用拆。
大部分组件走这条路。
STEP 1.1
HTML 里有 data-component 占位？
data-component = 底板上的预留卡槽
🔲 积木拼好了，但底板上有个卡槽标着"这里插 Button/Icon"。
流水线工人 _resolveDataComponents() 看到卡槽 → 去零件库取对应积木插上。

例：NavBar 的 preview_html 里：
<div data-component="💙 00.05_Button / Icon" style="width:40px;height:40px"></div>
↑ 这就是预留卡槽。渲染时自动填入 Button/Icon 的真实积木。
▼ 没有 preview_html？往下找
STEP 2
有 component_ref？单引用
取一块积木 → _resolveRef() 去零件库找
STEP 3
有 component_refs？套餐
照说明书拼模型 → 遍历零件清单，每块各调一次 _resolveRef()
▼ 连引用都没有？自动生成 fallback
STEP 4
有 _composed_preview_html？slots 拼装
📋 有零件编号 + 尺寸规格表 — slots 里精确记录着每块零件的编号（cid: Button / SearchBar）和尺寸（css: 40×40 / flex:1）。
流水线拿着编号去零件库取真正的积木，按规格拼好 → 出来的模型能交互（chip 能切子组件 variant）。

关键：零件编号是确定的，不是猜的 → 零误差拼装。
来源：generate.py 读 slots 规格表，逐个查零件库，拼成完整模型。
STEP 5
有 _tree_preview_html？图层翻译过渡期
📸 照 X 光片描轮廓 — 没有零件编号也没成品，但 Figma 插件导出时自动拍了张"X 光片"（_tree：每个图层的类型 + 位置 + 尺寸）。
流水线机械复印：FRAME→div，TEXT→span，RECTANGLE→div，每个像素位置都准 → 但它不知道哪块是 Button 哪块是背景，只知道"这里有个 40×40 的方块"。
外形准，不能交互。

⚠️ 这是过渡期应急：等设计师补上真正的积木（preview_html），这个"X 光复印件"就会自然淘汰。
STEP 6
都没有？CSS 骨架
🔲 空底板 — 只知道底板多大（width/height）、横着还是竖着（flex-direction），但上面没有积木。

看到这个 = 提示设计师"该补积木了"。

🔄 _resolveRef(ref) — 去零件库取积木的递归过程

1. 查 COMPONENTS_DATA[cid]
在零件库里找这块积木。找不到就查 COMPONENT_REF_MAP（别名货架）。
2. 确定要哪个款式（variant）
props 描述特征 告诉仓库你要什么 — "我要白色的、iPhoneX 款的"。仓库拿货架逐个比对：
  A 款：白色 ✓ iPhoneX ✗ → 1 分
  B 款：白色 ✓ iPhoneX ✓ → 2 分 ← 最高分，选它
  C 款：白色 ✗ iPhoneX ✗ → 0 分

单变体组件 没传 props 但货架上只有一款 → 直接拿唯一的那块
3. 取到 variant → 看它是什么类型
成品块 preview_html → 拼上，递归结束
又指向别的积木 component_ref → 再去取一轮
又是子模型 component_refs → 展开零件清单，逐块再取
↩ 回到 Step 1，直到所有积木都变成成品块（Leaf）为止
